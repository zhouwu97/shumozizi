# 题目解析、路线比较与连续建模

这一阶段要产出能指导实现的数学方案，而不是审批申请。

1. 分解所有必答问题、问题依赖、数据字段/单位/尺度和题意歧义。区分题目要求的对象、约束、目标和输出，不预先假定模型。遇到“总、同时、至少一个、分别”等词时，显式比较逐实体求和、时间并集、全体交集、加权或多目标等候选语义，逐一写出公式、单位和题面语言依据；程序与 oracle 一致只能验证给定公式的执行，不能证明公式就是题意。
2. 先做结构预检：不变量、界、极端情形、可消元变量、可分解结构、事件边界、小规模 oracle 与主要敏感性风险。给出至少一个题目特定的核心推导或反例，不把几何、物理或统计判断交给通用运行时。
3. 依据风险按需读取最多 3 个能力包：一个主能力包、至多一个交叉能力包、至多一个验证/不确定性包。可用 `skills/2analysis-modeling/SKILL.md`、`knowledge/cards/`、`knowledge/model-selection-matrix.md` 或一个匹配 Cookbook；资产只提供候选，不替代当前题的路线比较。
4. 生成两到三条真正不同的候选路线并进行比较。每条说明假设、可回答的问题、最低成本 probe、代价、风险和何时切换 fallback。先执行能区分路线的 probe，再确定主路线。
5. 有搜索、仿真或非光滑目标时，先确认已知可行例能够被题目特定精确评分恢复；再把候选生成、精确评分和搜索挑战实现为相互独立的职责。主报告说明目标、约束、覆盖/最优性理由和失败边界；脚本负责冻结详细输入输出。既有质量协议只在路线比较后约束实际执行证据。只有实际实施时才按需查阅 `docs/CODEX_WORKFLOW.md`。
6. 写入 `reports/ANALYSIS_MODELING_REPORT.md`：

   ```markdown
   # 赛题分析与建模方案
   ## 题目目标、数据与歧义
   ## 结构预检与核心推导
   ## 候选路线与最低成本 probe
   ## 主路线、fallback 与各问模型
   ## 实现接口、验证计划与最高风险
   ```

7. 将采用解释、主路线、fallback、放弃路线和下一步压缩写入 `state/DECISIONS.md`。正式题面在进入能力路由前还要由只读题面、禁止联网的新 Codex 对话做目标语义预审；若独立判断冲突，回到本阶段修正，不以公开同题答案定案。若简单模型已直接回答题目并通过必要验证，不为“使用高级方法”增加复杂度。
