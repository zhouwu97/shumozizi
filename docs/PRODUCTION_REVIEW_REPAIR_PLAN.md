# 生产优先与五层独立审核修复计划

## 1. 目标与范围

本计划修复当前项目“验证器强、生产器弱”的根本问题，并建立分阶段、独立、可收敛的论文审核链。

目标不是继续增加 JSON、封条或手填检查表，而是连续产出：

- 题意正确、路线可辩护的模型规格；
- 能从干净输入复现的代码、结果和图表；
- 与实验同步形成的完整论文；
- 经独立审核达到“国二稳定、国一可冲”门槛的提交包。

本计划不把 Codex 调度器、后台队列或自建多 Agent 框架引入仓库。审核对话由 Codex 桌面任务编排，仓库只保存审核请求、回执和证据。

## 2. 不变原则

1. `state.json` 仍是运行状态唯一来源，不能用聊天历史或任务 ID恢复进度。
2. 作者任务是唯一可以修改模型、代码、结果和论文的任务。
3. 审核任务默认只读，只能写自己的审核报告和回执。
4. 每个审核请求必须绑定当前 `revision`、配置锁、输入、代码、结果、图表和 PDF 哈希。
5. 阶段审核有主责范围，但任何审核都拥有发现致命跨域问题的否决权。
6. 审核意见必须证据化、可修复、可验证，不以报告长度或问题数量衡量严格程度。
7. 全面盲审可以循环，但必须有轮次上限、通过条件和人工升级出口。
8. AI 使用审查采用披露合规和写作风险等级，不把不可靠的“AI率百分比”作为硬门槛。

## 3. 目标工作流

```text
S0 题面解析与人工路线确认
  -> S1 建模规格生产
  -> R1 独立建模审核
  -> S2 编码、实验与问级论文同步生产
  -> R2 独立实验复现审核
  -> S3 全文组装与 Claim-Evidence Map
  -> R3 独立论文逻辑审核
  -> R4 格式、视觉与提交包审核
  -> R5 全面盲审循环
  -> G3 人工最终核包
  -> COMPLETE
```

R1-R4 发现阻断问题时返回作者任务修复；R5 发现问题时进入全面修复循环。文字和排版小修不重新执行完整 R5；模型、数字、核心图表、假设、约束或结论改变时必须重新执行受影响实验，并创建新的 R5 任务。

路线确认阶段额外冻结 `problem/PROBLEM_MANIFEST.json`，作为题目全集和必做问题的唯一来源。状态推进、逐问验收、论文生产和最终核包均通过 `verify_run_integrity()` 复验该清单及其题面哈希。

## 4. 五类审核任务

### R1 建模审核

输入：题面、附件摘要、问题简报、路线锁、模型规格和验证计划。

重点确认：

- 是否逐问理解题意，单位、范围和强制输出是否完整；
- 现实对象到变量、参数、目标和约束的映射是否正确；
- 主模型是否适配数据结构，基线是否廉价且公平；
- 目标函数、约束、变量域和算法是否可计算；
- 验证设计是否有能力推翻错误模型；
- 子问题之间的依赖是否成立。

结论：`ACCEPT`、`ACCEPT_WITH_MINOR_FIXES`、`SPEC_REVISION_REQUIRED`、
`ROUTE_REAPPROVAL_REQUIRED`、`BLOCKED_MISSING_INPUT`。

### R2 实验复现审核

输入：官方输入、清洗代码、执行清单、执行记录、结果、图表和模型规格。

重点确认：

- 是否有目标泄漏、训练评价污染或不公平数据划分；
- 指标、单位、基线和比较口径是否一致；
- 约束、收敛、稳定性、随机种子和失败样本是否复核；
- 代码是否能从干净输入重新生成当前结果；
- 图表是否由当前结果生成，坐标、标签、单位和数据是否一致；
- 结果支持的结论边界是否被正确限制。

结论：`REPRODUCIBLE`、`REPRODUCIBLE_WITH_WARNINGS`、`BLOCKED`。

### R3 论文逻辑审核

输入：题面、锁定模型规格、已接受结果、论文源文件、PDF、图表和 Claim-Evidence Map。

重点确认：

- 每个子问题是否有直接答案；
- 变量、假设、公式、算法和停止条件能否脱离代码理解；
- 关键数字能否定位到结果、表或图，并且结论没有超出证据；
- 基线、验证、限制和实际意义是否写清楚；
- 图表是否服务比较、诊断、机制或决策，而不是装饰；
- 是否出现内部生产术语、无来源数字、虚构引用或空泛拔高。

结论：`READY_FOR_COMPREHENSIVE_REVIEW`、`MAJOR_REVISION`、`NOT_READY`。

### R4 格式、视觉与提交审核

输入：最终 PDF、Typst/LaTeX 源文件、图表、支撑材料和当届官方规则。

确定性检查包括：

- A4、页边距、摘要专用第一页、正文页数和文件大小；
- PDF、源文件和支撑包中的匿名信息；
- 引用、参考文献、附录源程序、支撑材料列表和 AI 使用披露；
- PDF 哈希、结果哈希、图表哈希和提交包一致性。

视觉检查包括：

- 逐页渲染后的字体、缺字、遮挡、裁切、重叠、空白和可读性；
- 公式、上下标、符号和公式编号是否正确显示；
- 图表 caption、正文引用、单位、图例和数值是否对应。

结论：`COMPLIANT`、`FIX_REQUIRED`、`NOT_COMPLIANT`。

### R5 全面盲审

R5 由全新 Codex 对话执行。审核任务不接收作者解释、前四轮报告或上一轮修复意见，只读取当前冻结的完整提交和原始题目。

任务目标是独立判断：

- 模型、公式、算法、实验和结论是否整体正确；
- 图、表、字体、公式渲染和论文可读性是否达到正式提交标准；
- 论文是否完整回答题目，证据链能否从结论反向闭合到输入；
- 是否存在评委会导致降档的遗漏、夸大、错误、不可复现或格式问题；
- 综合水平对应什么奖项区间，以及判断置信度。

R5 不允许修改任何生产产物，只输出全面报告和评级。

R5 采用双轴联合结论：A 轴只由确定性完整性验证支持 `A_PASS` 或 `A_BLOCKED`；B 轴由独立评审给出 `B_STRONG`、`B_PASS`、`B_WEAK` 或 `B_REBUILD`，总分至少 75 且题目覆盖、模型深度、实验验证均至少 60 才算质量通过。程序强制检查四种联合结论的一致性：`FINAL_CANDIDATE`、`QUALITY_REPAIR`、`INTEGRITY_REPAIR`、`FULL_REPAIR`。

## 5. R5 收敛规则

每一轮使用新的 `review_round_id` 和新的审核对话。流程为：

```text
冻结 revision
  -> 创建 R5 独立任务
  -> 全面复现、渲染和评审
  -> 输出评级与问题
  -> 作者修复
  -> 按影响规则重跑
  -> 冻结新 revision
  -> 创建新的 R5 任务
```

默认标准：

- 单轮出现 P0/P1，必须修复；
- 评级低于 `B：国二稳定、国一可冲`，必须修复；
- 连续两轮独立 R5 达到 B 或以上，且无 P0/P1，才允许进入人工最终核包；
- 竞赛模式最多 2 轮，训练模式最多 5 轮；
- 达到上限仍不通过，状态为 `NOT_READY_FOR_SUBMISSION`，不得伪装成通过；
- 只有文字、错别字或纯排版修订，才允许做差异复核而不重新开完整 R5；
- 模型、数字、核心图表、假设、约束或结论变化，必须重新执行受影响实验并新开 R5。
- 任一非通过报告自动生成带来源哈希的 `REPAIR_PLAN.json`，逐条列出轴、受影响阶段、文件、预期改进和必需重测；路线漂移时仍须返回人工路线确认点。

评级采用：

```text
A  国一竞争力
B  国二稳定、国一可冲
C  国二边缘
D  省奖水平
E  尚不具备正式提交质量
```

评级必须附证据、置信度和降级原因。未经过历史获奖论文盲评校准前，评级只能标记为专家估计，不能宣传为稳定预测。

## 6. 独立新对话交接

每个阶段完成时，作者任务先写入审核请求，再由 Codex 桌面创建新的只读审核任务。请求至少包含：

- `run_id`、阶段、问题编号和 `review_round_id`；
- 当前状态 revision；
- 配置锁、题面、代码、输入、结果、图表和 PDF 哈希；
- 当前阶段审核 Skill；
- 允许读取路径和唯一报告输出路径；
- 结论枚举、预算和超时；
- 不得修改生产文件的限制。

审核结束后写入 `review_report.json` 和 `review_receipt.json`。作者任务只接受哈希匹配、状态匹配且报告结构合法的回执。

项目代码不启动 Codex、不管理后台队列、不依赖任务 ID恢复。若桌面任务创建能力不可用，状态停在 `WAITING_*_REVIEW`，保留可人工领取的请求。

## 7. 生产器修复顺序

### P0 结果正确性

- 撤销当前包含目标泄漏的 `q1-primary`；
- 在结果准入前加入目标/特征角色检查、训练评价检查和模型族最低验证要求；
- 要求 accepted 结果与路线锁、模型规格和执行记录一致；
- 为统计、优化、预测、仿真和分类建立最低证据规则。

### P1 生产器

- Route Producer：题意拆分、路线候选、基线、风险和预算；
- Model Specification Producer：变量、假设、公式、约束、算法和验证计划；
- Experiment Producer：baseline、primary、robustness/ablation、失败样本和图表；
- Paper Producer：从模型规格建立章节骨架，并随着结果增量更新。

### P2 状态与审核交接

- 将单线状态扩展为总体阶段加问级 model/experiment/paper 轨道；
- 增加审核请求、报告、回执和 revision 绑定；
- 将 R1-R5 变成桌面任务编排点，而不是 CLI 命令；
- 将重复审核、旧报告失效和影响范围复核写入状态不变量。

### P3 论文与合规

- 采用 2.5 cm 页边距、摘要专用第一页和强制分页；
- 增加公式、表格、参考文献、完整源程序、支撑材料列表和 AI 使用披露；
- 生成无内部 Evidence/Manifest/Ledger 术语的正文；
- 增加逐页视觉检查、字体和文件大小检查。

### P4 真实验收

至少完成三类陌生题：统计/数据题、优化题、机理或仿真题。每题从原始题面开始，不能使用题目专用现成代码。验收重点是最终完整论文，而不是两页 E2E 演示稿。

## 8. 需要修改的仓库位置

- `AGENTS.md`：从“一次最终自审”改为有预算的 R5 独立盲审循环，保留禁止无限调参和无限审核；
- `.agents/skills/mathmodel-review/`：拆分为 R1-R5 五个 Skill；
- `src/shumozizi/workflow/`：增加审核请求、回执和 revision 绑定；
- `src/shumozizi/results/sealing.py`、`workflow/validation.py`：增加语义准入和路线一致性；
- `schemas/`：增加审核请求、审核报告、审核回执和奖项评级结构；
- `templates/`：增加模型、实验、论文、综合审核模板和正式论文模板；
- `tests/`：增加目标泄漏、旧回执失效、盲审循环收敛、PDF 视觉缺陷和影响范围复核测试；
- `docs/CODEX_WORKFLOW.md`：更新生产优先主链和五层审核流程。

## 9. 完成定义

本计划完成不以“测试通过”单独判定，而必须同时满足：

1. 一道完整陌生题能够从题面走到完整论文；
2. R1-R4 有独立报告，且审核任务未修改生产文件；
3. R5 至少两轮独立盲审达到 B 或以上，或明确记录为未达标；
4. 任意核心数字都能从论文反向追溯到当前运行结果；
5. 公式、图表、字体、可读性和提交格式完成实际 PDF 检查；
6. AI 使用完成合规披露，不以虚构 AI 率代替质量判断；
7. 复现命令、环境和最终提交包在干净目录中可运行；
8. 人工最终核包前，状态和所有审核回执均通过哈希与 revision 校验。
