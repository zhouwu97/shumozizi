# 二、这次真正做对的地方

## 1. 时间区间算法已经实质修复

首次红队指出旧算法可能漏掉同一网格单元内的双根窄区间。现在改为基于显式 Lipschitz 上界的自适应区间分类：

* 固定烟幕活跃集合；
* 使用 (303\ \mathrm{m/s}) 作为时间方向裕量函数的保守 Lipschitz 常数；
* 单元可确认有效时计入下界；
* 可确认无效时排除；
* 不能确定时继续细分；
* 最终同时返回估计值、下界和上界。

这一设计在数学上是合理的。烟幕中心速度为3 m/s，导弹端点速度为300 m/s，有限线段集合的 Hausdorff 位移不超过300 m/s，距离函数和后续的 `min/max` 运算不会增大 Lipschitz 常数。

* 单元内部窄区间；
* 左端窄区间；

* 左端点有效；

# 四、Q5的30.552769是否正确

## 1. 求和算术正确

当前分量：

[

[

[

[

]

## 2. 但这只是一个建模口径

官方题面只要求对M1、M2、M3实施干扰，使有效遮蔽时间尽可能长，并没有给出跨导弹的标量聚合公式。

当前选择：

[
J_{\mathrm{sum}}
================

\mu(U_{M1})+\mu(U_{M2})+\mu(U_{M3})
]

是合理的，它衡量累计“导弹·秒”。

但是A196采用的是：

[
J_{\mathrm{all}}
================

\mu(U_{M1}\cap U_{M2}\cap U_{M3}),
]

即三枚导弹同时被遮蔽的时间，并得到21.0770秒。

所以：

* 30.55277导弹·s；
* 21.0770秒；

单位、意义和目标都不同，不能直接比较大小。

## 3. 当前30.55方案的三弹同时遮蔽只有0.076秒

我根据当前 `q5.json` 的完整区间重新计算了三种指标：

[
J_{\mathrm{sum}}
=30.55276934\text{ 导弹·s},
]

[
J_{\mathrm{any}}
================

\mu(U_{M1}\cup U_{M2}\cup U_{M3})
=23.48885032\text{ s},
]

[
J_{\mathrm{all}}
================

\mu(U_{M1}\cap U_{M2}\cap U_{M3})
=0.07637486\text{ s}.
]

唯一三重交集区间为：

[
[14.22629279,\ 14.30266765].
]

也就是说，当前方案虽然累计干扰时间高，但几乎从不同时遮蔽三枚导弹。

这不是代码错误，而是目标函数的自然结果：

> 等权求和倾向于把资源分散到各自最容易得到时长的位置，不主动要求三个时间集合对齐。

## 4. 最稳妥的处理不是只选一种口径

最终论文至少应同时报告：

[
J_{\mathrm{sum}},\qquad
J_{\mathrm{any}},\qquad
J_{\mathrm{all}},\qquad
\min(D_{M1},D_{M2},D_{M3}).
]

更重要的是，应独立求两套Q5方案：

* 最大化累计导弹·秒；
* 最大化三弹同时遮蔽时间。

然后给出二者的差异，甚至构建：

[
\max\left(
J_{\mathrm{sum}}
+\lambda J_{\mathrm{all}}
\right)
]

或Pareto前沿。

仅在当前求和方案上“顺便算一下交集”，不能回答A196那种同步防护需求。

# 八、连续时空证书：方法是进步，但当前区间太宽

当前空间证书采用：

* 7742个圆柱表面样本；
* 覆盖半径上界：

[
\rho=0.31622036\text{ m};
]

* 将有效烟幕半径从10 m缩为：

[
10-\rho=9.68377964\text{ m}
]

得到连续下界；

* 原10 m半径下的离散表面判定给出上界。

这个思路是合理的。

对于外部观察点和凸圆柱，通向任意内部点的视线会先经过圆柱近侧表面；若通向该表面点的视线已经被烟幕截断，那么通向内部点的更长线段也会被同一烟幕截断。因此表面覆盖可以推出实体覆盖，但论文必须把这个凸性和射线包含关系写清楚。

## 当前主要问题是证书宽度

| 问题 |                连续时空证书 |      宽度 |
| -- | --------------------: | ------: |
| Q1 |      [1.315, 1.393] s | 0.078 s |
| Q2 |      [4.430, 4.588] s | 0.158 s |
| Q3 |      [6.402, 6.403] s | 0.001 s |
| Q4 |    [10.999, 11.499] s | 0.499 s |
| Q5 | [29.625, 30.568] 导弹·s |   0.943 |

Q4和Q5的连续空间不确定性仍然很大。

因此：

* `30.55276934`只能叫高精度离散估计；
* `[29.625,30.568]`才是当前连续证书；
* 不能把6位小数写成物理精度。

论文应报告：

> 高精度估计30.553导弹·s，连续时空证书为[29.625,30.568]导弹·s。

正文最好保留到0.001或0.01，内部JSON继续保留更多位。

# 九、质量协议仍然严重高估了搜索充分性

当前 `results/quality.json` 把Q1–Q5全部标为：

```text
search_adequacy = passed
result_role = accepted
paper_allowed = true
```

这与实际证据不匹配。

## Q1–Q4的质量候选池过于形式化

Q1–Q4的质量候选只有三个合成标签：

```text
baseline
selected
perturbed
```

其中：

* `selected`就是当前生产结果；
* `perturbed`通常只是删除最后一个动作或使用一个手工扰动；
* 并没有导入真实搜索产生的大量候选；
* 也没有审查多个seed之间的排序、代理误差和局部稳定性。

所以它能证明：

* 当前结果能被重算；
* 当前结果满足约束；
* 一个简单扰动不如当前结果。

它不能证明搜索充分。

## Q5质量协议也只是复算已选代表

Q5候选池是：

[
2\text{个seed}\times16\text{个动作数}=32
]

个代表候选。

这些候选不是各动作数的完整搜索轨迹，而是主搜索已经为每个动作数选出的一个最优代表。质量协议再使用同一个 `model_core` 重算它们。

这可以证明结果文件没有篡改，但不能独立挑战：

* 主搜索漏掉的候选；
* 代理筛掉的高质量解；
* 不同参数化；
* 不同搜索器；
* 更大的连续变量覆盖。

因此当前质量结论应改为：

```text
execution_valid = true
feasibility_valid = true
exact_recomputed = true
search_adequacy = limited / unproven
```

至少Q3不能是 `paper_allowed=true`。
