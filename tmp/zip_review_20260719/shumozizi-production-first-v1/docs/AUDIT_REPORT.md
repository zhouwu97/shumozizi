# `shumozizi` / `shumoziyong` 建模与论文工作流审核报告

## 一、仓库事实

目标仓库 `zhouwu97/shumozizi` 当前是初始化仓库，尚无工作流、论文模板或执行代码。实际可审查的成熟内容位于 `zhouwu97/shumoziyong`，本报告以其当前 `main` 的 Contest v2、2024-C 示例、论文准入和独立评审链为对象。

## 二、总判断

当前问题不是单纯的“审核限制太多”，而是两类问题叠加：

1. **生产器缺失**：系统具备验证、结果绑定、打包和审查能力，但没有真正可执行的建模与论文生产编排。
2. **审查进入关键路径过深**：11 项逐问 Paper Admission、学习资产治理、全新 Reviewer 复审等机制适合赛后审计，不适合普通比赛主链。

更准确地说：

> 当前系统能够证明一份稿件哪里不合格，却没有足够机制主动产出一份高质量完整论文。

若做诊断性权重，主要矛盾约为“生产能力缺失占大头，审查过重占次要但明显的部分”。这不是统计比例，而是架构判断。

## 三、值得保留的能力

### 1. 数值真源与论文绑定

`result.json → verification.json → result_ledger.json → results.typ → PDF` 的方向正确。它能减少手抄数字、旧结果混入正文和修改后不一致。

### 2. 工程状态与论文质量分离

把工程验收、论文质量和最终提交分开，是对旧 Gate 体系的重要修正。代码跑通不能等于论文通过。

### 3. 独立验证边界

求解代码不能自行写 `verified=true`，结果变化使验证过期，这些规则应保留。

### 4. 支撑材料打包

排除密钥、缓存、审查草稿并打包代码、结果和图表，适合继续复用。

## 四、根本问题

## 4.1 CLI 是验证器，不是生产器

当前 Contest v2 公开命令只有：

```text
init
status
verify
package
```

`init` 创建的 `model.md`、`run.py`、`paper.typ` 和 `check.md` 都只是 TODO。架构文件还明确排除了 Agent 调度、LLM 调用和模型路线自动选择。

因此：

- 题意拆解无人执行；
- 主模型与基线无人选择；
- 实验计划无人形成；
- 图表与论文论证无人同步；
- CLI 只能等待外部任务把内容填好。

这就是为什么即使工程链越来越完整，最终仍可能只是几页技术报告。

## 4.2 审核在补偿生产缺陷

Paper Admission 每问固定 11 项，并要求每项提供定位证据。学习上下文还要求：

- 至少选择一个已授权规则或模式；
- 每问六类章节材料全部 READY；
- 每个学习资产填写采用位置或拒绝理由；
- PDF 和学习上下文摘要绑定。

这些检查能发现缺口，但不能创造缺失的模型、结果解释和论文段落。

实际风险是：

- 作者花时间填写证明表；
- 审核器检查字段完整；
- 论文主体仍然短、薄、缺少推导；
- 形式上的 PASS 依赖人工自填内容，而非真正的语义理解。

## 4.3 Paper Admission 与 Final Reviewer 重复

Paper Admission 已逐问检查题目回应、数学表达、算法、结果、基线、验证、图表、解释和边界；Final Reviewer 又按相近维度重新审查。

二者应当分工：

- 前置只做最小完整性检查；
- 终审负责质量判断。

当前两层都做“完整论文质量审查”，导致重复劳动。

## 4.4 写论文开始太晚

当前文档顺序仍然接近：

```text
分析建模
→ 编码实验
→ 完整论文写作
→ 工程验收
→ 准入
→ Reviewer
```

真正高质量论文应当从建模阶段同步生成：

- 建模时写变量、假设、公式和理由；
- 实验时写算法、参数、图表和结果解释；
- 验证暴露模型缺口后反向修正文稿。

等实验全部结束再写，最容易形成“结果汇总型技术报告”。

## 4.5 测试验证工程一致性，不验证建模与写作能力

当前测试重点覆盖：

- JSON 合法性；
- 摘要与 stale；
- Ledger 可重建；
- Typst 数值绑定；
- PDF 是否能编译；
- ZIP 排除规则；
- Paper Admission 字段完整性。

这些测试很重要，但不能证明：

- 陌生题能否被正确拆解；
- 模型是否针对题意；
- 实验是否有有效基线；
- 论文是否形成完整论证；
- 结果是否有决策价值。

必须新增真实陌生题生产验收，而不是继续增加字段测试。

## 五、当前论文模板的参赛风险

以 `contest_v2/examples/2024_C/main.typ` 为例。

### P0：页边距不符合 2026 官方规范

当前模板使用：

```typst
margin: (x: 2.1cm, y: 1.9cm)
```

2026 规范要求上下左右至少 2.5 cm。该问题应直接修复。

### P0：摘要页没有强制分页

电子版第一页必须是摘要专用页，正文应从下一页开始。当前模板在关键词后没有 `pagebreak()`，摘要较短时正文可能继续出现在第一页。

### P0：缺少合规附录

当前正文末尾只有“附件说明”，没有：

- 支撑材料文件列表；
- 建模使用的全部完整可运行源程序；
- 无程序或无支撑材料时的明确说明。

支撑 ZIP 中存在代码不能替代论文附录要求。

### P0：缺少 AI 使用披露链

当前工作流大量使用 Agent/LLM，却把重点放在“AI 痕迹风险审计”。官方要求关注的是透明披露：

- 正文相应位置标注；
- 参考文献列出工具；
- 支撑材料提供 `AI工具使用详情.pdf`；
- 包含工具版本、用途、关键交互、采纳和人工修改情况。

“看起来不像 AI”不能替代合规披露。

### P1：没有文件大小和身份信息检查

应检查：

- 论文电子文件不超过当年限制；
- 支撑材料不超过当年限制；
- PDF、代码、图表、元数据和路径中无姓名、学校、赛区等身份信息。

### P1：模板过于简陋

当前模板缺少稳定的：

- 公式编号与交叉引用；
- 图表标签与正文引用；
- 统一表格样式；
- 附录源代码样式；
- 参考文献管理；
- 摘要页控制；
- 字体跨平台策略。

这不会必然违规，但会明显降低完整论文的表达质量和可维护性。

### P1：2024-C 示例仍偏技术报告

示例的优点是数字谨慎、边界诚实、结果绑定清楚；缺点是：

- 每问高度压缩；
- 推导、算法伪代码和数据处理说明不足；
- 图表多用于结果陈列，机制解释仍有限；
- 统一模型与三问之间的递进可以更强；
- 附件、代码、AI 使用和复现说明未进入正式论文结构。

因此，工程验收通过不代表已经达到完整参赛论文水平。

## 六、正确的审核拆分

用户提出的顺序是对的，应改为：

```text
先审建模
→ 再审实验与复现
→ 再审论文逻辑
→ 最后审格式与提交
```

### 建模审查回答

- 是否理解题意；
- 变量、目标和约束是否正确；
- 主模型为何适配；
- 基线是什么；
- 验证计划能否推翻错误模型。

### 实验复现审查回答

- 代码能否从官方输入运行；
- 结果是否由当前代码生成；
- 数据划分和比较是否公平；
- 搜索、收敛、稳定性和误差证据是否足够；
- 哪些结论只能写成条件性结论。

### 论文逻辑审查回答

- 是否逐问明确作答；
- 模型是否能脱离代码独立理解；
- 结果、图表与结论是否一致；
- 是否解释“为什么”和“有什么意义”；
- 是否存在夸大、漏答和内部工程语言。

### 格式审查回答

- 是否符合当年官方格式；
- 是否匿名；
- 是否有引用、附录、源码和支撑清单；
- AI 使用是否完整披露；
- 文件大小和附件是否合规。

## 七、建议的仓库定位

### `shumozizi`

作为比赛生产主链：

- 人工确认题意和路线；
- Agent 建模、编码、实验、作图和并行写论文；
- 分层审查；
- 最终人工核包。

### `shumoziyong`

保留为：

- 旧题复盘；
- 争议复现；
- 能力评测；
- Result/Verification/Ledger 基础库；
- Formal Result、Manifest、Seal 和长期证据；
- 优秀论文学习治理实验。

不要再让 `shumoziyong` 的正式审计主链接管普通比赛。

## 八、最终结论

需要修改的不只是审核顺序。

真正有效的改造是：

1. 新增可执行的作者生产编排；
2. 论文从建模阶段并行写；
3. 先建模审查，再实验复现，再论文逻辑，再格式；
4. 普通比赛只保留四个硬检查点；
5. Paper Admission 从 11 项人工矩阵降为自动 claim-evidence map；
6. 只有模型、数字或核心结论改变时才完整复审；
7. 2026 合规模板必须立即修复页边距、摘要分页、附录源码、AI 披露、匿名和文件大小检查；
8. 旧重审计能力移到 `formal_audit`，不删除，但退出比赛关键路径。
